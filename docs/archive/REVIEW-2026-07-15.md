# Connectible - Code Review & KDE Connect Feature Parity (2026-07-15)

Date: 2026-07-15. Generated by two independent read-only reviews: a
full-codebase correctness/security/quality pass (covering the current
uncommitted working-tree diff plus a re-verification of every item in
[FINDINGS.md](FINDINGS.md), 2026-07-12), and a feature-by-feature
comparison against the real KDE Connect application's plugin set. No
code was changed by either review.

---

## 1. Code review

### Headline: most of FINDINGS.md is now fixed

The codebase has moved substantially since `FINDINGS.md` (2026-07-12).
Nearly every item that document lists as broken was independently
re-verified against the current tree and found resolved:

- Daemon lock-poisoning `.expect()` calls replaced with
  `unwrap_or_else(|poisoned| poisoned.into_inner())` everywhere except
  one `#[cfg(test)]`-only helper (`daemon/src/pairing/mod.rs:177`, not
  production code).
- Bounded input queue exists (`daemon/src/input/mod.rs:34`,
  `MAX_QUEUED_NON_MOVE_EVENTS`) -- the unbounded-queue DoS risk is gone.
- TLS key file TOCTOU fixed via `OpenOptions::mode(0o600).create_new(true)`
  (`daemon/src/tls.rs:98-106`), with a regression test.
- PIN modulo bias fixed via rejection sampling
  (`daemon/src/pairing/mod.rs:22-38`).
- Graceful SIGINT/SIGTERM shutdown is wired (`daemon/src/lib.rs:137-169`).
- `to_status()` now maps `ErrorCode` to distinct gRPC codes instead of
  collapsing everything to `Internal` (`daemon/src/grpc/service.rs:824-834`).
- Per-chunk resend on CRC32 mismatch is implemented and bounded
  (`daemon/src/grpc/service.rs:238-286`, T-306), not just an
  abort-only path.
- Wayland clipboard/input backends are real, substantial
  implementations (`daemon/src/clipboard/wayland_backend.rs`,
  `daemon/src/input/wayland_backend.rs`, ~500 lines each), not stubs --
  the "core promise" gap flagged as critical gap #1 in FINDINGS.md is closed.
- All six loopback-only RPCs (`DisconnectDevice`, `ForgetDevice`,
  `SubscribeLocalEvents`, `GetLocalState`, `SetRemoteInputEnabled`,
  `SetClipboardSyncEnabled`) correctly fail-closed via `require_loopback`
  (denies on missing peer addr) and each has a dedicated
  non-loopback-rejection test (`daemon/src/grpc/service.rs:1176-1244`).
- Desktop: `HomePanel.tsx`'s "Quick Actions" now genuinely navigate;
  `handleDisconnect`/new `handleForget` call the daemon and refresh;
  `SettingsPanel.tsx` uses the typed `ipc` wrapper and
  `errorCodeMessage()` everywhere -- no `as any`, no swallowed errors;
  `platformLabel()` is routed through `t()`; no raw
  `yellow-*/blue-*/green-*` Tailwind literals remain in non-test
  `.tsx` files.
- Mobile: the dead "Actions" tab and stale bidirectional-pairing
  comments are gone; the file-receive collision bug is fixed with
  transfer-id-keyed partial paths and directory-scanned unique final
  names (`mobile/lib/src/state/file_transfer_model.dart:271-390`, T-108).
- Deletion of `mobile/lib/src/widgets/radar_painter.dart` and the
  corresponding CSS/Tailwind keyframes leave no dangling references
  (only a stale, non-load-bearing comment remains at
  `desktop/src/styles.css:157`).

No correctness bugs, no injection/auth-bypass issues, no unsafe
deserialization, and no newly-introduced silent error swallowing were
found anywhere in the reviewed diff or the daemon/desktop/mobile code
paths it touches.

## 2. KDE Connect feature parity

Connectible's own README describes it as "a KDE Connect alternative."
The table below checks Connectible's actual code (not just planning
docs) against the real KDE Connect app's plugin set.

| # | Feature | Status | Evidence/notes |
|---|---|---|---|
| 1 | Clipboard sync (bidirectional) | Present, partial | `ClipboardData` message + `daemon/src/clipboard/mod.rs` (echo-suppression via hash, X11/Wayland backends), mobile `clipboard_model.dart`, desktop `ClipboardPanel`. Text-only -- proto comment: "MVP only guarantees text/plain; other types are best-effort." |
| 2 | File/URL/text sharing | Present (files); partial (text/URL) | `FileTransferStart`/`FileChunk`/`FileChunkRequest` fully implemented with resume+CRC32/SHA-256 (`daemon/src/transfer/mod.rs`), desktop `TransferPanel.tsx`, mobile `transfers_screen.dart`. No OS share-sheet/intent integration; no dedicated "share link" flow. |
| 3 | Notification sync (mirror/reply/dismiss) | Present but non-functional end-to-end | `NotificationData` proto + daemon plumbing + desktop `NotificationsPanel.tsx` exist (desktop is explicitly read-only/one-way), but mobile never sends any `NotificationData` -- `mobile/lib/src/state/device_list_model.dart:71-75` explicitly dropped `"notifications"` from advertised capabilities. No Android `NotificationListenerService`. No reply/dismiss-sync anywhere. |
| 4 | Remote input / virtual touchpad | Present & working | `RemoteInputEvent` proto, `daemon/src/input/{backend.rs,wayland_backend.rs}` (ydotool/X11/native Wayland), mobile `remote_input_screen.dart` (drag-to-move, tap/double-tap, two-finger scroll, keyboard with modifiers). Desktop-only injection target; Windows/macOS explicitly a non-goal. |
| 5 | MPRIS / media remote control | Entirely absent | No media-player code anywhere; not mentioned in any planning doc. |
| 6 | Presentation remote / laser pointer | Entirely absent | No matches anywhere. |
| 7 | Find my phone / ring device | Entirely absent | No RPC, message, or UI; not mentioned in planning docs. |
| 8 | Battery report | Present but non-functional end-to-end | `BatteryStatus` proto + `daemon/src/status/mod.rs` + desktop `StatusBar.tsx` exist, but mobile never constructs/sends `BatteryStatus` (same capability-removal comment as #3) -- display pipeline exists, never receives real data. |

| 11 | Telephony (SMS, missed calls) | Entirely absent | No sms/telephony code; not referenced anywhere. |

### Summary

Connectible implements exactly the subset of KDE Connect that
`PLAN.md`'s "Goals (MVP / v0.1.0)" section commits to: mDNS discovery,
PIN pairing over TLS 1.3, bidirectional text clipboard, resumable file
transfer, phone->desktop remote input, and phone->desktop
battery/notification forwarding. Of those last two, the wire format
and daemon/desktop plumbing exist but are currently **dead in
practice** -- the mobile client explicitly never sends
`BatteryStatus`/`NotificationData` (capabilities dropped, not just
unimplemented UI). If closing that gap is in scope, it's the single
highest-leverage feature move: no new proto messages needed, only a
mobile-side battery-plugin read + (for notifications) an Android
`NotificationListenerService` + permission/opt-in flow.

The other 11 KDE Connect plugins -- MPRIS, presentation remote,
find-my-phone, run-command, SFTP, telephony, contacts, airplane mode,
screen-lock sync, volume sync, connectivity report, remote-desktop
mirroring -- have **no code and no planning-doc mentions**. They are
out of scope by design (per `PLAN.md`'s stated MVP goals), not
deferred bugs.

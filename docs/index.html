01 — Capabilities

Everything a cable used to do. Nothing a cloud wants to.

Six capabilities, one protocol. Each is probed at daemon startup and advertised to peers, so the other side’s UI greys out what this machine can’t do — instead of failing when you try.

Shared clipboard

Copy on one device, paste on the other. Content hashing suppresses echo, so a synced copy never bounces back as a new one.

File transfer

Chunked with CRC32 per chunk and SHA-256 over the whole file, then renamed into place. Interrupted? It resumes from the highest contiguous offset.

Notification mirroring

Phone notifications appear on the desktop the moment they land — read them without reaching for the phone.

Remote input

Drive the desktop pointer and keyboard from the phone. Injected via ydotool/uinput and gated behind a capability flag — never silently half-working.

Battery status

The phone’s charge level and charging state, visible from the desktop status bar. A small thing you’ll miss everywhere else.

Zero-config discovery

Both peers advertise _connectible._tcp over mDNS and browse for each other. No IP addresses to type — and either side can initiate.

Capability honesty: capabilities ship inside each peer’s Identity, so a Wayland desktop without an input backend simply never claims remote_input. Features degrade at discovery time, not at use time.

02 — Pairing

Trust is established in thirty seconds, on your own screens.

No QR codes, no account linking, no email loop. One device shows a PIN, you type it into the other. That’s the entire ceremony — try it below.

Discover

Both devices advertise and browse _connectible._tcp over mDNS. The peer’s IP and port resolve automatically.

Connect

A TCP connection upgrades to TLS 1.3. The daemon’s rustls config is 1.3-only — an older handshake never completes.

Challenge

The responder generates a cryptographically random 6-digit PIN, valid for 30 seconds, and shows it on its own screen.

Confirm

You read the PIN off one screen and type it into the other. Comparison is constant-time; the third wrong attempt invalidates the PIN early.

Remember

Both sides persist the peer to their local SQLite devices table. From then on, reconnecting is a TLS handshake — no PIN again.

Pairing · live demo

This demo runs entirely in your browser and mirrors the real protocol: 30-second expiry (ERROR_CODE_PAIRING_TIMEOUT), three attempts, and a restart from Pair — never a ConfirmPin retry.

03 — Security

The connection is the security.

The lock isn’t a barrier in front of the product — it’s what makes every capability possible. And where the MVP cuts a corner, the corner is documented, not hidden.

TLS 1.3, mandatory

Every cross-device byte rides TLS 1.3. The rustls config offers no 1.2 fallback, so a downgrade isn’t a negotiation — it’s a refused connection.

No custom crypto

rustls and ring, plus standard hashing. Nothing hand-rolled, nothing clever — the most boring possible answer to every cryptographic question.

Constant-time PIN

PIN comparison leaks no timing information, PINs expire in 30 seconds, and the third wrong attempt invalidates the challenge entirely.

LAN-only by design

There is no relay to trust because there is no relay. If a packet would have to cross your router, it doesn’t get sent.

Known boundaries MVP

Documented, not hidden:

  • Certificate pinning lands in v1.0. Today, self-signed certs are accepted per connection — no trust-on-first-use yet.
  • The device database is plaintext SQLite. Encryption at rest is on the roadmap, not in the MVP.
  • Remote input is Linux/X11 today. On Wayland the remote_input capability is simply not advertised until the backend exists.
  • No internet relay. Off the LAN means offline — deliberately, but it’s still a boundary worth knowing.

04 — Architecture

A Rust daemon at the core. Thin, honest clients around it.

The wire contract is frozen as connectible.proto v1 — three codebases, one protocol, no drift.

Daemon · Rust

connectibled

One binary that owns discovery, TLS, the sync stream, and persistence. Generates its self-signed cert on first run; state lives under the XDG data dir.

tokiotonic · gRPC/HTTP2rustlsmdns-sdsqlx · SQLite

built + tested — 36 tests

Desktop · Tauri v2

desktop/

React UI over a webview-free Rust core crate that does all daemon-client work — so the logic builds and tests anywhere the daemon does.

Tauri v2React 18TypeScriptTailwind

built + tested — 3 e2e + 19 unit

Mobile · Flutter

mobile/

No local daemon — the app speaks the protocol natively and runs its own gRPC/TLS server, so pairing works in both directions: phone to desktop, desktop to phone.

FlutterDartProvider

code complete — needs a Flutter toolchain to build

One stream, eight frames

All real-time traffic multiplexes over a single bidirectional SyncStream RPC. Every message is one SyncFrame — a protobuf oneof — so adding a capability means adding a frame, not a connection.

Control-plane operations stay unary, where a clear request/response shape beats a stream:

Pair · begin the PIN ceremony ConfirmPin · finish it ListDevices · paired + discovered peers Ping · liveness
proto/connectible.proto 
message SyncFrame {
  oneof payload {
    ClipboardData     clipboard           = 1;
    RemoteInputEvent  input_event         = 2;
    FileTransferStart file_transfer_start = 3;
    FileChunk         file_chunk          = 4;
    BatteryStatus     battery_status      = 5;
    NotificationData  notification        = 6;
    Error             error               = 7;
    Identity          identity            = 8;
  }
}

rpc SyncStream(stream SyncFrame)
    returns (stream SyncFrame);

05 — Quick start

Clone, make, paired.

The Makefile is the front door — make with no target lists everything it can do.

terminal 
$ git clone https://github.com/lunanoir21/connectible-project.git
$ cd connectible-project
$ make setup   # one-time: frontend deps + system-dep check
$ make dev     # daemon + desktop app together; Ctrl-C stops both

# the rest of the front door
$ make test    # Rust workspace + frontend tests
$ make check   # clippy -D warnings + strict tsc
$ make build   # release: static daemon + frontend bundle

You’ll need

rustdaemon + desktop core (cargo)
node.jsdesktop frontend toolchain
protocprotobuf compiler, build-time only
webkit2gtk-4.1the desktop window, on Linux

Mobile builds need a Flutter toolchain — see mobile/README.md. Remote input on X11 additionally wants ydotoold and a udev rule; without them the capability just stays off.

Two tracks since 0.1.0: the roadmap phases in docs/TASKS.md (G through L below — N is real-device battery measurement and stays parked until requested) and a seven-phase audit-fix campaign in docs/TASKS-audit-fixes.md (X1–X7). No version bump yet.

Phase L — Clipboard rich content (images)

2026-07-24

Added

  • Clipboard sync now carries image/png payloads (new mime_type / binary payload fields on ClipboardData) alongside text, captured and applied on the daemon (Wayland wlr-data-control-unstable-v1 and X11), rendered as thumbnails with a "copy image" action on desktop, and round-tripped on mobile via super_clipboard.
  • A 10 MB hard cap on clipboard payload size, enforced identically on all three platforms; an oversized copy records a metadata-only history entry with a clear "too large" message instead of a silent drop or partial send.
Known gap: mobile's manual "Send" button still only reads the OS text clipboard when auto-monitor is off — only the always-on background poll captures images.

Phase K — Notification dismiss-sync

2026-07-23

Added

  • Dismissing a mirrored notification on desktop now clears the real notification on the paired phone (new loopback DismissNotification RPC, NotificationListenerService.cancelNotification on Android). The phone-to-desktop direction already existed; this closes the reverse one.
  • A per-notification-id echo guard on mobile stops the OS's own removal callback from bouncing back out as a spurious second dismiss frame.

Audit-fix campaign

2026-07-22/23

Three parallel read-only audits (desktop, mobile, docs) against the 0.1.0 codebase turned into a seven-phase fix campaign.

Fixed

  • Phone-initiated (QR/scan) pairing now persists the requester-side peer and pins the observed TLS fingerprint on ConfirmPin success, so a subsequent desktop-to-phone file push is no longer rejected as unpaired.
  • Received files on mobile are reliably reachable and no longer risk a crash on missing/renamed files; a UI-spawned desktop daemon no longer deadlocks once its stdout/stderr pipes fill.
  • Several silent failure paths now surface actionable, localized errors: pairing/connection failures on mobile's Home screen, a daemon-side FINGERPRINT_CHANGED rejection, restored transfer-history rows misreporting progress, and an externally-managed daemon's "stop" action.
  • PIN generation, cert/key file permissions, and unbounded flood/queue paths hardened further across both platforms.
  • Mobile now acquires an Android multicast lock while discovering and pauses mDNS sweeps while backgrounded.

Added

  • DiagnosticCheck.summary_key / .remediation_key give System Doctor text stable ids for client-side localization.
  • RecordTransferHistory / ListTransferHistory RPCs persist and paginate transfer history across daemon restarts.
  • PreArmPairingCode RPC lets desktop generate a pairing PIN up front to embed in a QR code.
  • Android foreground service for the receiving/discoverable role, keeping the inbound server + mDNS advertise alive under Doze/OEM background kills.

Changed

  • Docs reorganized under docs/; dead code removed (pre-audit doctor commands, orphan components, the UI-less "desktop drives the phone" input-session path).

Phase J — Persisted transfer history

Added

  • Daemon-owned transfer_history SQLite table serves both directions for desktop: incoming rows written directly by upload_file; outgoing rows reported back via a new loopback RecordTransferHistory RPC, since an outgoing send bypasses the local daemon entirely. A retention cap keeps the table bounded.
  • Mobile persists its own transfer history independently via shared_preferences, mirroring the existing paired-device-roster persistence pattern.
  • Transfer history now survives a daemon restart / app close on both platforms instead of losing every record.

Phase I — Retired the legacy chunk-transfer path

Removed

  • The original SyncStream-multiplexed FileChunk / FileTransferStart / FileChunkRequest frames and their daemon/desktop/mobile handling, superseded by the dedicated PrepareUpload / UploadFile streaming path (resume, cancel, progress, and whole-file hash verification already had parity there). SyncFrame proto fields 3, 4, 9 marked reserved.
Accepted trade-off: per-chunk corrupted-chunk resend has no equivalent on the dedicated path — a whole-file hash mismatch now forces a full restart instead of resending one chunk. TLS 1.3's AEAD already rules out transit corruption, so this only affects recovery efficiency for a rare sender-side fault.

Phase H — SQLite at-rest encryption

2026-07-20

Security

  • devices.cert_fingerprint is now encrypted at rest with AES-256-GCM (chosen over SQLCipher to avoid a C dependency / static-linking risk). Key sourced from a CONNECTIBLE_DB_KEY_FILE override, then the OS keyring, then a 0600 key-file fallback for headless hosts. Existing plaintext rows migrate in place on startup, each verified round-trip before being overwritten.
  • New System Doctor check reports which key source is active.

Phase G — Bidirectional mTLS identity pinning

2026-07-20

Security

  • The daemon now requests a client TLS certificate on every inbound connection and pins its fingerprint against the device_id on successful pairing — closing a gap where anyone on the LAN who learned a paired device's id could impersonate it without any credential. A later mismatch is rejected the same way a changed server fingerprint already was.
  • Desktop and mobile now present their own long-lived identity as their outbound TLS client certificate too.
Known asymmetry: a dart:io platform limitation means mobile's own inbound server can't yet perform the equivalent client-cert check. Documented, not silently dropped.

0.1.0

Connectible reaches a desktop-environment-agnostic, security-hardened, tested v0.1.0 across all three components (daemon, desktop, mobile).

Added

  • Native Wayland clipboard and remote-input backends, verified on Hyprland — no XWayland dependency for core sync features.
  • A per-chunk re-request mechanism for corrupted file transfers, implemented end-to-end with fault-injection tests (later retired in Phase I once TLS-level integrity made it redundant).
  • Bidirectional pairing: the mobile app runs its own gRPC/TLS server, so either a desktop or a phone can initiate pairing.
  • Forget/unpair device action, a pairable opt-out toggle, and remote-input/clipboard-sync toggles wired end-to-end on both platforms.
  • Systemd user-service packaging for the daemon.

Changed

  • Mobile's monolithic AppModel split into focused ChangeNotifiers, each with its own test coverage.
  • Desktop and daemon errors now map through a shared ErrorCode scheme instead of raw tonic::Status text.

Fixed

  • Wayland clipboard backend deadlock and a daemon shutdown hang, both found via manual verification with regression tests added.
  • PIN generation's modulo bias, a cert/key file-permission TOCTOU window, and an unbounded input-event queue.
  • TLS 1.3 enforcement gap on the mobile server, mobile file-receive filename collisions.
